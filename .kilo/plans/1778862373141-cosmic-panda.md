# Plan: supabase-classlist and rename-json-files

## TL;DR

Two new skills (each with a slash command and Python script):
1. **supabase-classlist** — syncs `docs/students.txt` → Supabase `classlists` table via `supabase-py`
2. **rename-json-files** — renames `local-working/{folder}-{student_id}.json` → `{student_id}.json`, validated against Supabase classlists via `supabase-py`

Both use **only supabase-py** (`from supabase import create_client`). No psycopg2, sqlalchemy, or raw REST calls.

> **Note:** The PyPI package is `supabase` (v2.28.3 installed), not `supabase-py` (that name is deprecated/removed from PyPI). Import path is `from supabase import create_client, Client`.

## Students.txt file format (validated)

```
Line 1 (header):  Class,student_ID, name    ← single tab field, commas are literal
Lines 2+:         M2-4A<TAB>36018<TAB>Pooh<TAB>0
                  ^class  ^id    ^name ^ignored trailing column
```

- **93 data rows**, all with exactly 4 tab-separated columns
- All `student_id` values are 5-digit numeric strings — no duplicates across the file
- 5 classes: `M2-4A`, `M2-5A`, `M3-3A`, `M3-4A`, `M3-5A`
- Trailing `0` in column 4 is ignored (not included in upsert)
- Header is skipped entirely — the commas in `Class,student_ID, name` are literal text, not delimiters

**Remote Supabase table (existing state):**
- 113 rows already present (from other classes like `Yr 11 Summer 2026`, plus some overlap with M2/M3 classes)
- Columns: `student_id` (PK), `name`, `class`, `created_at`, `updated_at`, `user_id`
- Upsert correctly handles: new students inserted, existing students' `name`/`class` updated, other classes untouched

**Parsing verdict:** No changes needed. The current `csv.reader(f, delimiter="\t")` with header-skip and trailing-column-ignore pattern handles every row correctly.

## Step 2: Create supabase-classlist script

**`src/supabase_classlist.py`**

- Uses `from supabase import create_client` — the ONLY DB interface
- Reads env vars: `SUPABASE_URL`, `SUPABASE_ESL_KEY` (service_role key for full table access)
- Parses `docs/students.txt` (tab-separated: header skipped, 4 data columns — class, student_id, name, trailing `0`)
- For each row, builds dict: `{"student_id": "36018", "name": "Pooh", "class": "M2-4A"}`
- Upserts each row into `public.classlists` via `supabase.table("classlists").upsert(row).execute()`
- Upsert is safe: handles new students (insert) and class transfers (update on PK conflict)
- Reports counts: upserted, errors
- Supports `--check <student_id>` flag: returns exit code 0 if found in classlists, 1 if not found (used by rename-json-files workflow)

**Existing remote data:** 113 rows already present (Yr 11 Summer 2026 + partial M2/M3 classes). Upsert only touches rows with matching `student_id` PKs — other classes left untouched.

**Test file `tests/test_supabase_classlist.py`**:
- Test parsing of students.txt (tab-separated lines, header skip, trailing column ignored)
- Test data row construction
- Test `--check` mode logic (mock supabase response)
- No live Supabase calls in tests — mock the client

## Step 4: Create rename-json-files script

**`src/rename_json_files.py`**

- Reads all `.json` files from `local-working/`
- For each file, extracts `student_id` from JSON content
- Queries Supabase classlists via **supabase-py** to verify student_id exists:
  ```python
  result = supabase.table("classlists").select("student_id").eq("student_id", sid).execute()
  exists = len(result.data) > 0
  ```
- If exists: renames file to `{student_id}.json` (flat in `local-working/`)
- If not found: prints alert with file path and student_id, skips renaming
- Reports: renamed count, skipped count, list of skipped files
- Handles: duplicate student_ids across files (warn), missing student_id key, empty directory, file-in-use errors

**Test file `tests/test_rename_json_files.py`**:

- Mock supabase client for existence checks
- Test rename logic with mock filesystem
- Test edge cases: missing key, duplicate IDs, Supabase down, no files

## Key decisions

1. **supabase (supabase-py) only** — no psycopg2, sqlalchemy, raw REST. `from supabase import create_client` is the sole DB interface. (PyPI package name is `supabase`, GitHub repo is supabase-py.)
2. **`SUPABASE_ESL_KEY`** is the service_role key (`sb_secret_...`) — used for write access to classlists. The `SUPABASE_ACCESS_TOKEN` (pat token) is not used by supabase-py.
3. **Upsert, not insert** — handles both new students and class transfers. Supabase detects conflicts on `student_id` primary key.
4. **Script for rename-json-files** — an agent workflow (like git-backup) could loop and rename, but a Python script is more testable, handles edge cases better, and uses supabase-py consistently. The command just runs the script.
5. **Flat output** — renamed files go to `local-working/{student_id}.json` (no subfolder), matching the user's specification.
