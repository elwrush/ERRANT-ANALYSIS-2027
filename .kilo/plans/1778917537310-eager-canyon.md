# Research Plan: Writing Error Analysis by Cohort

## 1. Sample Sizes

| Cohort | Students | Records (pre-filter) | After word_count >= 40 | Sampling |
|--------|----------|---------------------|------------------------|----------|
| M2 | 36 | 289 | ~287 (2 dropped) | All students |
| M3 | 36 | 396 | ~391 (5 dropped) | Random from 61, seed=42 |
| **Total** | **72** | **685** | **~678** | |

Sampling plan: `local-working/sampling_plan.json` (seed=42, reproducible).

## 2. Cohort Assignment Logic

- **M2 cohort**: Student IDs present in Supabase `classlists` table with current class M3-4A or M3-5A → they submitted writing when they were in M2 the previous academic year
- **M3 cohort**: Student IDs in `student_submissions` (skill=Writing) but NOT present in `classlists` → left/graduated the programme, submitted when they were in M3

## 3. Name and Class Handling

- **M2 students**: Name and class retrieved from `classlists` Supabase table. Class recorded as M2 (their level when submitting), not their current M3-4A/M3-5A label.
- **M3 students**: Not in classlists. Referred to by student ID only (no name available). Class recorded as M3.

## 4. HTML Cleaning (Pass-Fail System)

55% of sampled records contain `<br>` tags (3,133 occurrences in 685 records; no other HTML tags present).

**Cleaner:**
- Replace `<br>` (any case, optional `/`) with newline `\n`
- Strip any remaining HTML tags via regex `<[^>]+>`

**Pass-fail criteria:**
- PASS: cleaned text has word_count >= 40 AND no remaining HTML-like content (`<` not present in cleaned text)
- FAIL: word_count < 40 after cleaning → record excluded from analysis

## 5. Student-level Aggregation

A student may have multiple writing records (avg 8 for M2, 11 for M3). All records for a student are:
1. Cleaned individually (HTML stripped)
2. Combined into one `student_text` (joined with `\n\n`)
3. Total `word_count` = sum of individual record word_counts
4. Fed as a single text to ERRANT analysis (one correction run, one summary)
5. Inserted as ONE row in `error_reports` per student

This avoids pseudoreplication (Gries 2021) — student is the unit of analysis.

## 6. ERRANT Pipeline

**Uses the existing skill unchanged** — `src/errant_analysis.py` via `--batch research` mode.

| Setting | Value |
|---------|-------|
| Correction model | `gpt-4o-mini` (OpenAI direct) |
| Summary model | `gpt-4o-mini` (OpenAI direct) |
| Pass 1 temperature | 0.1 |
| Pass 2 temperature (double-check) | 0.3 |
| Summary temperature | 0.8 |
| Parallel workers | 5 |
| Double-check | Intersect edits from both passes |
| Overcorrection threshold | >3 tokens flagged |

## 7. What Gets Written to error_reports

**One row per student** (72 rows total: 36 M2 + 36 M3).

| Column | Value | Source |
|--------|-------|--------|
| `student_id` | e.g. `"29579"` | student_submissions |
| `class` | `"M2"` or `"M3"` | Derived from cohort logic |
| `name` | Name string or `""` for M3 | classlists table (M2 only) |
| `error_percent` | e.g. `12` | Total corrections ÷ total words × 100 |
| `summary` | LLM-generated paragraph | Summary model at temp 0.8 |
| `word_count` | Total words across all records | Sum of record word_counts |
| `r_noun`, `r_spell`, ... (45 columns) | Integer count per error type | ERRANT error classification |
| `m_det`, `u_verb`, ... etc. | Integer count | See full column list in SKILL.md |

The `insert_error_reports` function in `errant_analysis.py` needs `word_count` added to the row dict (line ~303-308).

The existing pilot for student 29579 (Ka-nhom) confirmed: 12 records → 1,330 words → 156 errors (12%) → 32 error type columns populated.

## 8. Implementation Steps

1. **Create `src/research_prep.py`**: Fetches records from `student_submissions` for all 72 sampled students. Cleans HTML. Filters word_count >= 40. Combines per-student. Writes one JSON per student to `outputs/research/{student_id}.json` (format: `student_id`, `student_text`, `word_count`, `name`, `class`).

2. **Update `src/errant_analysis.py` `insert_error_reports`**: Add `word_count` to the row dict so it's written to `error_reports`.

3. **Run ERRANT pipeline**: `python src/errant_analysis.py --batch research` — processes all files in `outputs/research/` with 5 parallel workers. Each `<student_id>.json` gets corrected, ERRANT-analyzed, summarized, and inserted to `error_reports`.

4. **Verify**: Check `error_reports` for 72 rows with `word_count` populated and accurate.

## 11. Model Migration: OpenRouter → Direct OpenAI API

### Why

OpenRouter rate-limits gpt-4o-mini heavily (~35s per correction call, 5 workers, 62 students → 20+ min). Direct OpenAI API typically serves gpt-4o-mini at 2-3x the speed (~15s per call).

### Key Constraints

- `.env` file has `OPENAI_API_KEY=sk-proj-MF...` (OpenAI project key, 164 chars)
- System has stale `OPENAI_API_KEY=sk-or-v1-...` in process env (OpenRouter key)
- System has `OPENAI_BASE_URL=https://openrouter.ai/api/v1` globally set (affects OpenAI SDK, NOT raw requests)
- No `.env` file existed previously — one was created with the OpenAI key
- `load_dotenv()` does NOT override existing env vars by default, so `os.environ.get("OPENAI_API_KEY")` reads the stale process value

### Fallback

If the OpenAI project key consistently returns 401:
- The key may need an `OpenAI-Project` header — extract project ID from platform.openai.com → Settings → Project
- The key may lack `gpt-4o-mini` access — check [platform.openai.com/limits](https://platform.openai.com/settings/organization/limits)
- Billing may not be set up — check [platform.openai.com/billing](https://platform.openai.com/settings/organization/billing/overview)
- **Alternative model**: If OpenAI key is unreachable, fall back to `deepseek-chat` via `DEEPSEEK_API_KEY` (already verified at 1.4s response time)
