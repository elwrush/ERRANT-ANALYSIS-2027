### 2. Reducing ERRANT "uncategorised" errors

The ERRANT classifier (read from source) returns `"OTHER"` when edits don't match its rule-based framework. Key causes found in `errant/en/classifier.py`:

- **Complex multi-token edits** that ERRANT can't decompose (e.g. idiom replacements)
- **Derivational morphology** where similarity falls below thresholds
- **Word choice errors** where a content word is replaced by another (ERRANT tags as POS-based, not lexical)
- **String similarity edge cases** where Levenshtein distance falls between thresholds (0.55–0.8)

Mitigation strategy:

1. **Prompt engineering**: Tell the model to make MINIMAL corrections — one error at a time, preserving as much of the original text as possible. Avoid rephrasing.
2. **Post-processing**: Before ERRANT, split complex multi-token corrections into atomic edits where possible.
3. **Categorize OTHER**: After ERRANT, run a fallback classifier on OTHER edits using string similarity and POS tags to assign best-guess categories.

## Workflow

### 0. Agent asks for input files

The agent must ask the user for precise file paths. These are the JSON files output by `ingest-images` (e.g., `outputs/class_a/12345.json`). The agent should:

1. List available `outputs/` subfolders
2. Ask user to specify one or more file paths
3. Confirm the paths are correct before proceeding

### 1. Load and split input

For each file (contains `student_id` and `student_text`):
- Read the JSON
- Split `student_text` into sentences (using spaCy sentence segmentation)
- Store original sentences for pairing

### 2. Generate corrected text

Use the AI model to correct the full `student_text`. Prompt engineering is critical:

```
You are a grammar correction tool. Correct the following student essay.
Correct ONLY grammatical errors (verb form, subject-verb agreement, articles,
prepositions, spelling). Make the MINIMAL change needed — one error at a time.
Do NOT rephrase, improve style, or change vocabulary. Preserve all capitalisation
and punctuation as-is unless correcting an error.

Original text:
{student_text}

Corrected text (return ONLY the corrected text, no explanation):
```

This prompt minimizes complex edits, which helps ERRANT classify errors.

### 3. Run ERRANT analysis

ERRANT takes original + corrected text as input:

```bash
errant_parallel -orig <orig_file> -cor <cor_file> -out <m2_file>
```

Or use the Python API:

```python
import errant
annotator = errant.load("en")
orig = annotator.parse(original_text)
cor = annotator.parse(corrected_text)
edits = annotator.annotate(orig, cor)
```

Extract from edits:
- Error type (`edit.type`)
- Original span (`edit.o_str`)
- Corrected span (`edit.c_str`)

### 4. Compile error summary

Group edits by error type. For each type, collect:
- `type`: e.g., `R:VERB:SVA`
- `count`: number of occurrences
- `example`: one representative original→corrected pair (e.g., `"go → goes"`)

### 5. Generate corrected text with markup

Using ERRANT edit offsets, insert `<u>` tags around corrected spans in the original text. Walk through the text token-by-token, wrapping modified spans.

### 6. Calculate error rate

```
error_rate = round((number_of_corrections / word_count) * 100)
```

Where `number_of_corrections` is the count of non-noop ERRANT edits, and `word_count` is the number of tokens in the original text.

### 7. Write output

Save to `/local-working/{input_folder}-{student_id}.json`:

```json
{
  "student_id": "12345",
  "original_text": "He go to store every day and buy milk.",
  "corrected_text": "He goes to the store every day and buys milk.",
  "sentence_pairs": [
    {"original": "He go to store every day and buy milk.", "corrected": "He goes to the store every day and buys milk."}
  ],
  "errant_analysis": {
    "errors": [
      {"type": "R:VERB:SVA", "example": "go → goes", "count": 1},
      {"type": "M:DET", "example": "store → the store", "count": 1},
      {"type": "R:VERB:FORM", "example": "buy → buys", "count": 1}
    ],
    "uncategorised": []
  },
  "corrected_with_markup": "He <u>goes</u> to <u>the</u> store every day and <u>buys</u> milk.",
  "error_rate": 30
}
```

# Corrective Actions: LLM-GEC + ERRANT Limitations

Based on research into ERRANT's classifier source, the "Adapting LLMs for Minimal-edit GEC" paper (Staruch et al., 2025), "Edit-level Majority Voting" (Goto et al., 2025), and the Bryant et al. (2017, 2023) GEC surveys.

## Research Findings Summary

The ERRANT classifier is a **rule-based system** with hardcoded POS-tag rules, dependency-label patterns, string-similarity thresholds (0.55 for spelling, tuned on FCE+W&I), and dictionary lookups. Anything unmatched returns `OTHER` or `R:OTHER`.

LLMs present four specific problems for ERRANT-based pipelines:

| Problem | Source | Effect on our pipeline |
|---------|--------|----------------------|
| **Over-correction** | LLMs modify grammatically correct text to improve fluency | Produces false-positive edits; harms ERRANT F0.5 (which weights precision 2×) |
| **Rephrasing vs minimal editing** | LLMs generate fluent rewrites instead of atomic token edits | Multi-token edits fall through to `R:OTHER` in ERRANT's classifier |
| **Formatting noise** | LLMs add markdown, bold, commentary ("Okay, the corrected sentence is:") | Creates spurious `U:SPACE` / `R:ORTH` edits in ERRANT alignment |
| **No noop handling** | ERRANT expects explicit "noop" annotations for sentences requiring no correction | Our output doesn't indicate when the model correctly identifies error-free text |

|-----------|--------|

## Gaps Remaining

### GAP-1: No overcorrection detection

The pipeline has no way to distinguish between legitimate corrections and over-corrections. Research shows LLMs can correct things that aren't errors, and we have no mechanism to flag or filter these.

**Evidence**: Goto et al. (2025) show majority voting catches over-corrections. Staruch et al. (2025) show error-rate adaptation during training is needed for precision control. We rely on prompt alone.

**Action**: Add an `overcorrection_warnings` field to the output JSON. Flag edits spanning more than 3 original tokens as potential overcorrections. This gives downstream consumers (human reviewers, report generators) a signal to inspect.

**Level of effort**: Low (~15 lines of code in `process_file`).
**Priority**: High.

### GAP-2: No noop / identity check

If the model correctly identifies that the essay needs no correction, it should return the original text unchanged. Our pipeline currently processes this through ERRANT anyway, producing either empty results or spurious edits from tokenization differences.

**Action**: Before running ERRANT, compare orig and cor texts (whitespace-normalized). If identical, set `error_rate: 0`, `errors: []`, skip ERRANT. Add `"needed_no_correction": true` to output.

**Level of effort**: Low (~10 lines in `process_file`).
**Priority**: High.

### GAP-3: No double-check pass (majority voting equivalent)

The "Edit-level Majority Voting" paper shows that generating multiple correction candidates and keeping only edits present in both runs significantly improves F0.5 by filtering out model hallucinations and overcorrections.

**Evidence**: Goto et al. use 5 candidates. We can approximate with 2 candidates at different temperatures.

**Action**: Add an optional double-check mode. The pipeline runs correction twice:
1. Pass 1: temperature=0.0 (most conservative)
2. Pass 2: temperature=0.3 (slightly more permissive)
Only keep edits found in BOTH runs. Edits found only in one run are flagged as "uncertain" in the output. This is a poor-man's majority voting without requiring ensemble models.

**Level of effort**: Medium (~40 lines, second API call + edit intersection logic).
**Priority**: Medium.

### GAP-4: No edit-width pre-splitting for ERRANT

ERRANT's classifier returns `OTHER` for multi-token edits it can't decompose. If the model produces a 4-token rephrase like "was not able to → couldn't", ERRANT sees one complex edit rather than individual changes.

**Evidence**: The ERRANT classifier explicitly returns `OTHER` for multi-token replacements at line ~300 of `classifier.py`: "Tricky cases. else: return 'OTHER'".

**Action**: After LLM correction, pre-split multi-token edits into atomic token-level edits before feeding to ERRANT. For each edit where len(o_toks) > 1 AND len(c_toks) > 1, attempt to decompose into smaller 1:1 or 1:0 edits using the same Damerau-Levenshtein alignment ERRANT uses internally.

**Level of effort**: Medium-High (~50 lines, requires understanding ERRANT's alignment internals).
**Priority**: Medium.

### GAP-5: No quality metadata in output

The current output JSON has no fields to help downstream consumers judge the quality of the model's corrections. A reviewer reading `error_rate: 12%` doesn't know if those are genuine errors or overcorrections.

**Action**: Add metadata fields to the output:
```json
{
  "metadata": {
    "model": "mistralai/mistral-small-3.2-24b-instruct",
    "temperature": 0.1,
    "identity_check": false,
    "overcorrection_count": 2,
    "uncertain_edit_count": 0,
    "total_edit_count": 8,
    "edit_width_stats": {
      "max_span": 3,
      "avg_span": 1.2,
      "multi_token_edits": 1
    }
  }
}
```

**Level of effort**: Low (~20 lines).
**Priority**: Medium.
