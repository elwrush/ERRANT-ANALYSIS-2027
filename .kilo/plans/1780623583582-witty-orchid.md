# Plan: Supabase Daily Backup + Writing Records Migration

## Goal

1. Set up a **daily automated backup** of the Supabase database using GitHub Actions + Supabase CLI (free tier)
2. Migrate all **writing assignment records** from `student_submissions` to `error_reports` with proper demographic data, M2/M3 enrollment class labeling, and a three-layer fallback safety net

---

| Item | Count |
|---|---|
| `student_submissions` total | 2,087 |
| Writing records (`skill = 'Writing'`) | 976 |
| Writing records with `student_text` | 973 |
| Existing `error_reports` | 142 |
| `classlists` entries | tracked in Supabase |

### Schemas (relevant columns)

**Source: `student_submissions`**
`id`, `student_id`, `submission_date` (timestamptz), `student_text`, `word_count`, `skill`, `academic_year`

**Reference: `classlists`**
`student_id`, `name`, `class` (academic level, e.g. "M3-4A")

**Dest: `error_reports`**
`student_id`, `class`, `name`, `error_percent`, `summary`, `word_count`, `academic_year`, `date`,
+ 45 error code columns (`r_noun`, `r_noun_num`, ..., `other`, `unk`)

### Class-label convention

- `class` field in `error_reports` uses M2/M3 as **enrollment status**, not academic level
- **M2** = student_id found in `classlists` (active/enrolled)
- **M3** = student_id NOT found in `classlists` (left the program)

## Phase 1: Daily Supabase Backup via GitHub Actions

### Step 1.1 — Obtain the DB connection string

Go to Supabase Dashboard → Project Settings → Database → Connection string → copy the `postgresql://postgres:[PASSWORD]@db.[REF].supabase.co:5432/postgres` URL.

### Step 1.2 — Add GitHub repository secret

Go to **GitHub → Repo Settings → Secrets and variables → Actions → New repository secret**:

| Name | Value |
|---|---|
| `SUPABASE_DB_URL` | `postgresql://postgres:[PASSWORD]@db.[REF].supabase.co:5432/postgres` |

### Step 1.3 — Create `.github/workflows/daily-supabase-backup.yml`

```yaml
name: Daily Supabase Backup

on:
  schedule:
    - cron: '0 2 * * *'   # 02:00 UTC daily
  workflow_dispatch:       # manual trigger from Actions tab

jobs:
  backup:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    env:
      SUPABASE_DB_URL: ${{ secrets.SUPABASE_DB_URL }}

    steps:
      - uses: actions/checkout@v4

      - name: Install Supabase CLI
        uses: supabase/setup-cli@v2
        with:
          version: latest

      - name: Backup roles
        run: supabase db dump --db-url "$SUPABASE_DB_URL" -f backups/roles.sql --role-only

      - name: Backup schema
        run: supabase db dump --db-url "$SUPABASE_DB_URL" -f backups/schema.sql

      - name: Backup data (COPY format)
        run: supabase db dump --db-url "$SUPABASE_DB_URL" -f backups/data.sql --data-only --use-copy

      - name: Clean dump (strip superuser-only lines that fail on restore)
        run: sed -i '/^ALTER OWNER/d; /^CREATE EXTENSION/d; /^ALTER DEFAULT PRIVILEGES.*OWNER/d' backups/schema.sql

      - name: Commit backup to repo
        uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: "backup: $(date +%Y-%m-%d)"
          file_pattern: backups/
```

### Step 1.4 — Test the workflow

1. Commit and push the workflow file
2. Go to GitHub → Actions → "Daily Supabase Backup" → "Run workflow"
3. Verify `backups/roles.sql`, `backups/schema.sql`, `backups/data.sql` are committed
4. If it fails, check the runner logs — common issues:
   - Wrong `SUPABASE_DB_URL` format (must use `postgresql://`, not `postgres://`)
   - Network timeout (add `--no-verify-ssl` flag if using IPv6)
   - Docker not available on runner (unlikely on `ubuntu-latest`, but switch to `apt install postgresql-client` + raw `pg_dump` as fallback)

### Step 1.5 — Verify backup persists

After 2 successful daily runs, verify that `git log --oneline -- backups/` shows commits on consecutive days. The daily `pg_dump` connection also keeps the Supabase free project from being evicted (7-day inactivity timer reset each day).

### Step 2.3 — Migration script logic (`src/migrate_writing_records.py`)

**Inputs:**
- `local-working/backup_error_reports_2026-06-05.json` (Layer 1, written in Step 2.1)
- Supabase connection

**Logic:**

1. **Load classlist membership set**
   ```python
   classlist_ids = {row['student_id'] for row in client.table("classlists").select("student_id,name").execute().data}
   ```

2. **Load classlist name lookup**
   ```python
   classlist_names = {row['student_id']: row['name'] for row in ...}
   ```

3. **Build dedup set from error_reports**
   ```python
   existing = client.table("error_reports").select("student_id,date").execute().data
   existing_pairs = {(r['student_id'], str(r['date'])) for r in existing if r['date']}
   ```

4. **Fetch all writing records**
   ```python
   records = client.table("student_submissions").select("student_id,submission_date,student_text,word_count,academic_year").eq("skill","Writing").neq("student_text","null").execute().data
   ```

5. **For each record, determine and insert:**
   ```python
   for r in records:
       sid = r['student_id']
       date_str = r['submission_date'][:10]  # "2025-09-11" from ISO 8601
       if (sid, date_str) in existing_pairs:
           skipped += 1
           continue

       row = {
           "student_id": sid,
           "class": "M2" if sid in classlist_ids else "M3",
           "name": classlist_names.get(sid, ""),
           "summary": r['student_text'],      # temporary — overwritten later by ERRANT analysis
           "word_count": r.get('word_count', 0),
           "academic_year": r.get('academic_year'),
           "date": date_str,
           "error_percent": None,
       }
       # Set all 45 error code columns to 0
       for col in ERROR_CODE_COLUMNS:
           row[col] = 0
       batch.append(row)
   ```

6. **Batch insert** (100 rows per batch to avoid timeouts):
   ```python
   client.table("error_reports").insert(batch).execute()
   ```

**Expected results:**
- ~973 writing records fetched
- ~142 skipped (already in error_reports — pre-existing matches)
- ~831 new rows inserted
- Total error_reports after migration: ~973 (142 existing + 831 new)

**Edge cases:**
- `submission_date` is timestamptz — extract date with `[:10]` (YYYY-MM-DD)
- `word_count` may be NULL — default to 0
- `student_text` may be empty string — still insert, summary will be ""
- `academic_year` may be NULL — leave as None/NULL
- "writing" (lowercase) records (2 found) — include in query with `eq("skill","writing")` or use `.in_(["Writing","writing"])`

## Phase 3: Batch ERRANT Analysis — Error Counts Only (no summaries, no reports)

### Goal

For the ~844 records we just migrated (those with `error_percent IS NULL`), run the correction + ERRANT annotation pipeline to populate **only** the 45 error code columns and `error_percent`. No summaries, no Typst markup, no local files, no reports.

### Identification

| Condition | Value |
|---|---|
| Records to process | `error_percent IS NULL` in `error_reports` (= our 844 migrated records) |
| Each record's student_text | Look up from `student_submissions` by matching `(student_id, date)` → `(student_id, submission_date::date)` |
| Output | Upsert only `student_id`, `date`, `error_percent`, and the 45 error code columns |

### What we import from `errant_analysis.py`

| Import | Purpose |
|---|---|
| `correct_text()` | Calls DeepSeek V4 Flash for text correction |
| `pre_split_edits()` | Post-processes ERRANT edits (splits R:OTHER multi-token edits) |
| `classify_edits()` | Groups edits by ERRANT type, returns `[{type, count}, ...]` |
| `ERRANT_CODE_TO_COLUMN` | Maps ERRANT types (e.g. `"R:VERB:TENSE"`) to column names (e.g. `"r_verb_tense"`) |
| `TEMPERATURE` (= 0.6) | Correction temperature |
| `CORRECTION_PROMPT` | Prompt template |

### What we do NOT use (skip)

- `build_corrected_typst()` — Typst markup (not needed)
- `align_sentences()` — sentence pairs (not needed)
- `generate_summary()` — LLM feedback (not needed)
- `_build_output()`, `write_output()` — local JSON files (not needed)
- `lookup_student_info()` — demographic data already populated from migration
- `insert_error_reports()` — uses `insert()`; we need `upsert()` on `(student_id, date)`

### Core pipeline per record

```python
# 1. Correction (calls DeepSeek V4 Flash, temperature 0.6)
corrected_text, _, _ = correct_text(original_text, nlp)

# 2. Noop shortcut
if original_text.strip() == corrected_text.strip():
    return {"error_percent": 0, **{col: 0 for col in ERROR_CODE_COLUMNS}}

# 3. ERRANT diff
orig_parse = annotator.parse(original_text)
cor_parse = annotator.parse(corrected_text)
edits = annotator.annotate(orig_parse, cor_parse)
edits = pre_split_edits(annotator, edits, orig_parse, cor_parse)

# 4. Classify and count
errors_list, _, _ = classify_edits(
    edits,
    orig_tokens=[t.text for t in orig_parse],
    cor_tokens=[t.text for t in cor_parse],
)

# 5. Map to column counts
counts = {col: 0 for col in ERROR_CODE_COLUMNS}
for eg in errors_list:
    col = ERRANT_CODE_TO_COLUMN.get(eg["type"])
    if col:
        counts[col] = eg["count"]

# 6. Error rate (NULL if word_count < 40)
correction_count = sum(eg["count"] for eg in errors_list)
error_rate = round(correction_count / word_count * 100) if word_count >= 40 else None
```

### New script: `src/batch_errant_upsert.py`

**Flow:**

1. Connect to Supabase
2. Query `error_reports` WHERE `error_percent IS NULL` → list of `(student_id, date, word_count)`
3. For each, query `student_submissions` for matching `(student_id, submission_date::date)` → `student_text`
4. Build join list: `[{student_id, date, word_count, student_text}, ...]`
5. Load spacy model + ERRANT annotator (once, reused)
6. **Loop** with progress bar:
   a. Run the core pipeline on `student_text`
   b. Build upsert row: `{student_id, date, error_percent, r_noun: 3, r_verb_tense: 1, ...}`
   c. Collect into batches of 100
7. Batch upsert via `client.table("error_reports").upsert(batch, on_conflict="student_id, date").execute()`
8. Report: `X processed, Y upserted, Z skipped`

### Upsert behavior

Only these columns are sent in the upsert row:

| Column | Values |
|---|---|
| `student_id` | (for conflict match) |
| `date` | (for conflict match) |
| `error_percent` | Calculated error rate |
| `r_noun` ... `unk` | 45 error code columns |

Columns **NOT** touched by upsert: `class`, `name`, `summary`, `word_count`, `academic_year`, `created_at`, `id`.

PostgreSQL `ON CONFLICT (student_id, date) DO UPDATE SET` only updates columns present in the INSERT clause. Unspecified columns keep their existing values.

### Edge cases

| Case | Handling |
|---|---|
| LLM timeout/error | Retry up to 3x (built into `@retry`). If all fail, skip record, log warning |
| Empty student_text | Skip record (shouldn't happen — filtered) |
| `word_count < 40` | Set `error_percent = NULL` (too short for reliable rate) |
| Identical original/corrected | Skip ERRANT, all counts = 0, `error_percent = 0` |
| `{student_id, student_text}` mismatch | Multiple submissions same student/date → use the first matching `student_text` |
| Record not in `student_submissions` | Skip with warning |

### Error handling strategy

- **Rate limit**: Add 0.5s delay between API calls
- **LLM failure**: Retries via `@retry(MAX_RETRIES=3)`, then skip with warning
- **Supabase upsert failure**: Retry the batch once; if still fails, save to `local-working/failed_batches.json`
- **Interrupt recovery**: Track last processed record; can resume by re-running (processed records now have non-NULL `error_percent`)
