1. Locate the `COPY "public"."error_reports" (...)` section in `backups/data.sql`
2. Parse the column header line to get field order
3. Parse the tab-separated rows between `FROM stdin;` and `\.`
4. Handle PostgreSQL COPY escaping (`\N` = NULL, `\t` = tab, `\\` = backslash, `\n` = newline)
5. Build a DataFrame with columns: `id`, `created_at`, `student_id`, `class`, `name`, `error_percent`, `summary`, `word_count`, `academic_year`, `date`, plus 45 error-type columns (`r_noun`, `r_noun_num`, ..., `unk`)

## Stage 2: Statistical Analysis

### 2.1 Overall Error Rankings (Section 1 of report)

**Method:**
1. Sum all 45 error-type columns across ALL 986 records
2. Normalise to errors per 100 words using total word_count
3. Rank by frequency (highest count first)
4. Produce a table with columns:
   - Rank
   - Error code (e.g., R:SPELL)
   - Plain-English meaning (e.g., "Spelling error")
   - Count (raw total)
   - Per 100 words (normalised rate)
   - Example input sentence (harvested from local-working)
   - Example corrected sentence (harvested from local-working)

**Critical check:** Does the top-3 align with what the outline expects? If not, I will flag this in the text — "While one might expect verb tense to dominate, the data actually shows..."

## Stage 4: Methodology Section (Section 5 of report)

Drawn from `archive/summary.md` and the pipeline's actual implementation:

1. **Data collection:** Handwritten essays → scanned → OCR (Gemini 2.5 Flash via OpenRouter)
2. **Correction:** DeepSeek V4 Flash with minimal-edit prompting (whole-text prompt, temperature 0.6)
3. **Error annotation:** ERRANT v3 rule-based classifier (55 error types across R/M/U categories)
4. **Summary generation:** DeepSeek V4 Flash at temperature 0.8
5. **Storage:** Supabase `error_reports` table

**Critical context I will add:**
- The LLM was explicitly prompted to make minimal corrections (not fluency rewrites)
- The data represents ONE academic year and ONE teacher's students
- The ERRANT tool was developed at Cambridge but the LLM correction is a modern enhancement

## Stage 5: Limitations Section (Section 4 of report)

### 5.1 Address from outline:

**LLM hallucination and over-correction:**
"I will note that LLMs default to fluency editing and may rewrite grammatically correct sentences. My analysis pipeline used minimal-edit prompting to mitigate this. Where available, I will report the overcorrection_rate from the data."

**The 'OTHER' error category:**
"ERRANT classifies some edits as OTHER when no rule matches. I will quantify what proportion of edits fell into this category. If it is small (< 5%), this is acceptable. If it is large, this IS a finding — it tells us our students make errors that ERRANT cannot classify, which may indicate L1-transfer patterns that merit further investigation."

**Generalisability:**
"My analysis is confined to M2-4A and M3 (3A, 4A, 5A) students taught by one teacher. The school has three grade levels (M1, M2, M3) each with skill levels A, B, C — nine cohorts in total. My results cover four of these nine. I cannot claim these findings generalise to M1 students, to B/C streams, or to other teachers. However, the error patterns I identify ARE likely representative of Thai secondary students at these levels, and the methodology is replicable."

### 5.2 Limitations I will add (not in outline):

- **Single academic year** — cannot assess whether error rates are improving year-on-year
- **OCR errors** — the ingestion model may introduce transcription errors that cascade into false corrections
- **No human-validated gold standard** — I cannot compute precision/recall of the automated corrections
- **ERRANT's English-centric taxonomy** — may miss error types specific to Thai-L1 English
